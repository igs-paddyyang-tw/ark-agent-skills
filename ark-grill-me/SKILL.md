# ark-grill-me

在實作前拷問設計——逐一走完決策樹的每個分支，直到達成共識。

## 觸發條件

- 「拷問」、「grill me」、「grill」、「stress test plan」
- 「質疑設計」、「挑戰方案」、「設計審查」
- 「review plan」、「拷問我的設計」
- 「在寫 code 之前先確認設計」

---

## 核心理念

> 你不是在面試。你是在對話。目標是把腦中模糊的設計意象，透過被提問的方式具體化。

Plan Mode 的問題：一次給出完整計畫 → 人容易說「OK」就通過 → 控制權被 AI 拿走。

Grill Me 的解法：AI 提問、人回答 → 人保有設計控制權 → 每個決策都被顯式確認。

---

## 提問規則

### 一次一個問題

每次只問一個問題。等使用者回答後，簡短確認（1-2 句），立即問下一題。

### 提供選項 + 推薦答案

每個問題附 2-4 個具體選項，標註推薦答案：

```
Q3：日報的輸出格式應該是什麼？

1️⃣ 單一 HTML 檔案（可直接瀏覽器開啟）⭐ 推薦
2️⃣ Markdown 檔案（輕量，適合 Git 版控）
3️⃣ HTML + PDF 雙版本
4️⃣ 其他（請說明）
```

選項要具體、反映真實可能的選擇。避免泛泛的「是/否」——除非問題真的是二元的。

### 能自己查的不問

如果問題可以透過檢查 codebase、檔案結構、或現有文件回答，自己去查，不要問使用者。

---

## 執行流程

### Phase 1：理解範疇

1. 收到使用者的設計主題
2. 檢查相關 codebase / 文件（如有）
3. 識別決策樹的主要分支

### Phase 2：逐一拷問

針對每個分支，從高層到細節逐步提問：

```
決策樹範例（每日新聞日報 Skill）：
├── 資料來源
│   ├── 哪些 RSS Feed？
│   ├── 更新頻率？
│   └── 失敗重試策略？
├── 輸出格式
│   ├── HTML / MD / PDF？
│   ├── 主題風格？
│   └── 是否支援多語言？
├── 觸發方式
│   ├── 排程 cron？
│   ├── 手動觸發？
│   └── 兩者都要？
└── 邊界條件
    ├── RSS 來源全掛怎麼辦？
    ├── 文章數量上限？
    └── 重複文章過濾？
```

### Phase 3：產出決策摘要

所有分支解完後，產出格式化摘要：

```markdown
## 決策摘要：{設計主題}

### 已確認決策

| # | 決策點 | 決定 | 理由 |
|---|--------|------|------|
| 1 | 資料來源 | Hacker News + TechCrunch RSS | 涵蓋中英文科技圈 |
| 2 | 輸出格式 | 單一 HTML（暗色主題） | 可直接瀏覽器開啟 |
| 3 | 觸發方式 | cron 每日 08:00 + 手動 /news | 自動為主、手動為輔 |
| ... | ... | ... | ... |

### 邊界條件

- RSS 全掛：fallback 到快取 + 標記「資料可能過時」
- 文章上限：每來源最多 10 篇，總計不超過 30 篇
- 重複過濾：以 URL 為唯一鍵，24 小時內去重

### 未決事項

- [ ] 是否需要 LLM 摘要（成本考量待確認）

### 下一步

此摘要可作為 `ark-superpowers` 產出 Spec 的輸入：
> 「根據以上決策摘要，幫我寫 spec」
```

---

## 品質檢核（Phase 3 前自動執行）

在產出摘要前，自我檢查：

- [ ] 是否有決策之間的矛盾？
- [ ] 是否有未覆蓋的邊界條件？
- [ ] 是否有隱含的依賴未被討論？
- [ ] 使用者是否有被動回答「OK」超過 3 次？（提示要主動參與）

如果檢核發現問題，追加 1-2 個補充問題再進入 Phase 3。

## 與其他 Skill 的串接

```
ark-grill-me（拷問設計）
    ↓ 產出「決策摘要」
ark-superpowers（產出 Spec）
    ↓ 產出 spec.md
ark-skill-creator（產出 Skill）
    ↓ 根據 Spec 實作
ark-code-spec-validator（驗證一致性）
    ↓ Drift Report
修復 → 重新驗證 → Ship
```

## 適用場景

| 場景 | 問題數 | 效果 |
|------|--------|------|
| 新 Skill 設計 | 8-15 題 | 釐清範疇、避免 scope creep |
| API 設計 | 10-20 題 | 釐清端點、資料模型、錯誤處理 |
| 架構決策 | 15-30 題 | 走完 trade-off、記錄 ADR |
| 重構計畫 | 10-15 題 | 確認影響範圍、遷移策略 |

---

## 反模式（避免）

| ❌ 反模式 | ✅ 正確做法 |
|-----------|------------|
| 使用者全部回答「OK」 | 主動質疑推薦答案，補充自己的想法 |
| AI 問太細的 UI 問題 | 高保真問題留給 prototype 階段 |
| 範疇太大（50+ 題） | 先拆解任務，對每個子任務分別拷問 |
| 拷問完直接寫 code | 先產出 Spec，再根據 Spec 實作 |

---

## 注意事項

- 每次只問一個問題，不要一次傾倒多個問題
- 推薦答案要有理由，不是隨便選一個
- 使用者回答「其他」時，追問細節再繼續
- 如果發現設計有嚴重矛盾，直接指出而非默默接受
- 問題數量通常 8-30 題，視範疇而定
- 語言：繁體中文（跟隨使用者語言）
